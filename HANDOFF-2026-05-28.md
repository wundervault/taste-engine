## What happened today, in order

1. **Acknowledged the v4 dish-gen still had flavor-gesturing problem.** User pushed back on dishes that named a flavor in their title without delivering it in the ingredient list (e.g., "Cauli-Style Sofritas Bowl" with no cauliflower). Root cause: prompt told LLM to "be inspired by trend X" but didn't gate on whether brand could deliver X. Plan: build deliverability matrix in code, route undeliverable signals to gap-fill section.
2. **Built `data/flavor_definitions.json`** — hand-curated map of each of 50 vocab terms to required ingredient signature (`match_terms`, `min_matches`). Conservative: "mole" needs literal "mole" SKU; "jerk" needs 2 of (allspice / scotch bonnet / thyme / "jerk"); fancy false positives like "mole" matching "guacamole" prevented via word-boundary regex with optional trailing 's' to catch plurals.
3. **Built `scripts/compute_pantry_fit.py`** — produces `data/pantry_fit.json` with per-(flavor, brand) deliverability. Results were harsh but accurate: Chipotle 3/50, CAVA 4/50, Sweetgreen 7/50 deliverable.
4. **Wired pantry-fit grid into Streamlit Pantry and Gaps tabs** — per-flavor row × per-brand column ✓/✗ matrix, per-brand gap callouts showing missing SKUs. Pantry tab also got a "delivers vs cannot deliver" side-by-side for each brand.
5. **User raised the data-pool question**: was the analysis using lunch-relevant restaurants or all restaurants? Currently it was all restaurants. Pivoted to **dual-pool design**:
   - Pool A (competitive): indie + lunch-open + $-$$ — what your customer is choosing instead today
   - Pool B (leading): indie + chef-driven (dinner-only OR $$$+) — what the chef-driven scene is popularizing for ~2 years out
   - Chains dropped from both via hand-curated denylist
6. **Built `scripts/tag_restaurants.py`** — classifies every gmaps restaurant by pool membership, adds new columns to `restaurants` table (`pool_competitive`, `pool_leading`, `is_chain`, `price_lower`, `opens_before_1pm`, `lunch_busyness`). Mutex pools by construction.
7. **Added `signal_ranking_dual()`** — returns both pool scores side-by-side per flavor.
8. **Cross-pool signal differentiation now visible**: e.g., Mission birria comp 0.49 / lead 0.24 ("Established" — mainstream taqueria culture), Mission truffle comp 0.22 / lead 0.37 ("Rising" — chef-driven, hasn't crossed over).
9. **User flagged that 27% of reviews were undated** (the `reviews_snippets` field — Google's algorithmic pull-quotes have no timestamp). And asked about chronological depth. **Pivoted to Bright Data's Google Maps Reviews product (`gd_luzfs1dn2oa0teb81`, NOT the Fast Maps Search dataset)** which returns paginated review feeds with `days_limit=N` for server-side recency filtering.

## What to NOT do

- **Don't re-trigger BD 730-day pulls without reason** — they redundantly fetch the 365-day data and BD bills you for both. Cost ~$20-30 per cycle. The current snapshot is sufficient.
- **Don't run `dish_generator.py` without the pantry-fit filter** — the old behavior caused systematic flavor-gesturing (Mole Bowl with no mole, Truffle Bowl with no truffle). The v5 prompt explicitly constrains via `pantry_fit.json`.
- **Don't pass city KEY where city LABEL is expected** — the bug caught this morning. Use `_city_label()` helper if writing new queries.
- **Don't conflate "pantry doesn't intersect current top trends" with "brand is failing."** Chipotle's "No current fit" grades reflect a vocab/pantry alignment gap, not company performance. Frame as opportunity, not failure.
- **Don't spawn Agent subagents without `pantry_fit.json` in their reading list** — they'll happily flavor-gesture too if not given the deliverability data.
- **Don't restore the `filter_lunch_competitors.py` path** — `tag_restaurants.py` superseded it. Both files exist but only the tagger is wired in.
- **Don't trust scores in the dashboard alone for non-statistical audiences** — use the strength labels (`strength_pill()` helper). Raw 0.92-style numbers are for analyst backup only.

## Demo argument, in one paragraph

Taste Engine measures the intersection between national flavor trends (DMA-level Google Trends) and local indie restaurant review signal (sub-pool: competitive lunch indies vs chef-driven dinner indies, both with chains excluded). For each of 3 brands × 3 neighborhoods, it produces two kinds of recommendations: ship-now dishes built honestly from current pantry (no flavor-gesturing — the deliverability matrix in code prevents it), and gap-fill SKU recommendations with operational-lift estimates and local comp restaurant context. The strongest single finding is Williamsburg jerk at "Very strong" on both pools (113 indie mentions) with no fast-casual chain able to deliver it from current pantry — a defensible, code-validated gap recommendation. Universal top gap across cities: mole, a single-SKU addition that opens premium menu tiers for all three brands.
